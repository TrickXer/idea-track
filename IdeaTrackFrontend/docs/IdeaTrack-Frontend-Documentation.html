1. Project Introduction

IdeaTrack is a role-based internal innovation platform where employees submit ideas, reviewers evaluate them in staged workflows, and administrators govern user access, categories, analytics, and proposal operations.

What the Frontend Does

  • Provides a complete browser-based interface for idea submission, review, approval, proposal creation, analytics, and profile management.
  • Applies role-aware navigation and route protection so each user sees only the surfaces relevant to their responsibilities.
  • Connects to a Spring Boot backend through REST APIs and server-sent events for real-time notifications.

Primary User Groups

  • Employees: create ideas, track status, manage proposals, and maintain their profiles.
  • Reviewers: review assigned ideas, collaborate in stage discussions, and submit decisions.
  • Admins: manage users, categories, reports, overdue reviewers, bulk operations, and system health.
  • Super Admins: manage the full user base including admins.

Business Problem Solved

The application replaces fragmented idea handling across spreadsheets, email, and manual approvals with a structured digital workflow. It centralizes user access, review assignments, notifications, analytics, and audit-friendly user operations in a single interface.

Role of the User Management Module

User Management is the operational control center for platform access. It allows admins and super admins to create accounts, assign roles, update user details, delete users, and configure reviewer-stage mappings that directly affect the review pipeline.

3. System Architecture

Frontend Architecture Pattern

The frontend follows a feature-oriented SPA architecture with route-driven pages, reusable components, and a thin global state layer. Most business data is fetched close to the page or feature component that needs it, while auth, notifications, and toast feedback are elevated into providers. 

Browser
  -> React App
    -> App.tsx providers
      -> AuthProvider
      -> NotificationProvider
      -> ToastProvider
      -> BrowserRouter
        -> ProtectedRoute
          -> Layout
            -> Feature Pages
              -> Components
                -> utils/* API modules
                  -> Spring Boot backend (REST + SSE)

Provider Layer

Auth, notifications, and toast feedback are initialized once near the application root and reused throughout the route tree.

Route Layer

Routes separate public pages from authenticated pages and apply role restrictions through a dedicated guard component.

Feature Layer

Pages assemble feature-specific components and call API modules directly rather than relying on a centralized data store.

User Management Workflow

  1. Admin navigates to /admin/users or super admin navigates to /super-admin.
  2. The page loads users from /api/admin/all and filters visible roles according to console type.
  3. Tabs segment the list by role. Search narrows results on name, email, user ID, and department.
  4. Creating a user opens RegisterUserModal, which loads departments, validates basic input, and posts to /api/admin/create.
  5. On success, the UI shows a temporary password popup and refreshes the user list.
  6. Selecting a user opens UserModal, which computes a field-level diff and submits only changed values to /api/admin/{id}.
  7. Deleting a user goes through ConfirmationModal, then calls DELETE /api/admin/{id}, refreshes the list, and emits toast feedback.
  8. The Stage Assign tab lets admins bind reviewers to department category stages, which directly influences reviewer workload and downstream review flow.

Role and Permission Model

The frontend enforces coarse-grained access via route guards and fine-grained operational limits through page configuration. For example, AdminConsole only allows creation and editing of reviewers and employees, while SuperAdminConsole additionally exposes admin-level role handling.

6. State Management

Where State Is Stored

  • AuthProvider: JWT token, decoded payload, roles, login, and logout.
  • NotificationProvider: live notification list, unread count, connection status, and singleton SSE connection lifecycle.
  • ToastProvider: transient UI feedback messages.
  • Feature-local state: user lists, forms, active tabs, modals, and loading flags remain inside pages and components.

Global State vs Local State

Global Local
Authentication, notification stream, toast queue Admin console tab, selected user, search term, modal open state, profile edit mode

Data Flow Pattern in User Management

Admin action
  -> component event handler
    -> utils/adminApi.ts request
      -> backend response
        -> local state refresh or modal close
          -> toast feedback
            -> rerendered user list

The project intentionally avoids Redux or Zustand. This keeps the state model simple but pushes more responsibility onto page components to refetch and synchronize their data explicitly after mutations.

7. API Integration

Axios Configuration

  • src/utils/restApi.ts defines a shared Axios instance with base URL http://localhost:8091.
  • Request interceptor adds Authorization: Bearer <jwt-token> when present.
  • Response interceptors log 401 and 403 cases; a separate app-level interceptor translates many backend errors into user-facing toasts.

8. Authentication & Authorization

Login Flow

  1. User submits email and password on the login page.
  2. AuthProvider.login() posts credentials to /api/auth/login.
  3. The backend returns a JWT token.
  4. The token is saved to localStorage under jwt-token.
  5. The token is decoded client-side to extract roles.
  6. The user is redirected to admin, reviewer, or employee landing routes based on highest role.

Protected Routes

ProtectedRoute blocks unauthenticated users and optionally checks whether at least one of the user’s roles appears in the route’s allow list.

How User Management Respects Access Control

  • /admin/users is limited to ADMIN and SUPERADMIN.
  • /super-admin is limited to SUPERADMIN only.
  • AdminConsole restricts role creation and editing to reviewer and employee accounts.
  • SuperAdminConsole expands this to include admin accounts.
Security caution: role checks are client-side UX gates and must still be enforced by the backend. The frontend correctly assumes the backend remains the source of authorization truth.

9. User Journey

  1. User opens the application and lands on a public marketing-style landing page.
  2. User signs up or logs in.
  3. After login, routing redirects based on role: employee dashboard, reviewer dashboard, or admin dashboard.
  4. Employees create ideas, view idea details, and eventually convert accepted ideas into proposals.
  5. Reviewers work from their dashboard, inspect assigned ideas, and submit decisions or stage discussions.
  6. Admins manage platform structure through user management, category management, bulk operations, reports, and health views.
  7. Users maintain personal details in Profile Hub, including password and profile photo.
  8. Notifications appear in real time through SSE and remain available in a paginated activity page.

11. Security

  • Token protection: JWT is stored in localStorage and automatically attached to API calls. This is simple but carries XSS exposure if the application were ever compromised.
  • XSS mitigation: the frontend mostly renders plain strings and does not rely on unsafe HTML insertion.
  • Input validation: profile and security forms include client-side validation; create-user flow uses minimal client checks and leans on backend validation.
  • Secure API usage: auth headers are centralized, and protected routes reduce accidental exposure of privileged screens.
  • Role-based security: user-management pages are visible only to authorized roles, while backend endpoints should remain the final enforcement layer.

Security Improvement Opportunities

  • Move API base URL to environment variables.
  • Add proactive token expiration handling instead of waiting for 401 failures.
  • Harden duplicate-email validation before user creation to improve UX.

12. Error Handling

Implemented Handling

  • Global Axios interceptor maps server errors into toast messages.
  • Feature components catch specific failures such as failed profile loads, failed user updates, or failed assignment removals.
  • Loading spinners and empty states exist on dashboards, notifications, assignments, and profile screens.

User Management Edge Cases Covered

  • Department fetch failure when creating users.
  • Save-without-changes prevention in UserModal.
  • Password policy validation before changing credentials.
  • Confirmation before destructive actions such as deleting users or profiles.

Edge Cases Still Mostly Backend-Driven

  • Duplicate email on user creation.
  • Permission denied after stale client role state.
  • Expired JWT during session.
  • Malformed or missing notification metadata.

13. How to Explain the User Management Module in an Interview

2-Minute Version

I built a role-based user-management flow inside a React and TypeScript SPA. Admins and super admins can create users, assign roles, update user details, delete accounts, and configure reviewer-stage assignments that drive the review workflow. Access is controlled through JWT-based auth, route guards, and role-aware UI composition. The module uses Axios for API calls, React Context for auth and notifications, and local component state for operational screens.

5-Minute Deep Explanation

The system separates concerns cleanly. App.tsx composes global providers and route protection. AuthProvider decodes JWT roles and exposes them through a hook, so navigation and protected routes can respond immediately to the user’s privileges. The admin user-management pages then specialize behavior further: AdminConsole only manages reviewer and employee accounts, while SuperAdminConsole can also manage admins. User creation and editing are modal-driven. RegisterUserModal loads departments, submits account creation, and surfaces the generated temporary password. UserModal computes a diff payload so unchanged fields are not resent. In parallel, the profile subsystem handles self-service updates, password changes, photo management, and activity history. ReviewerStageAssignment extends user management into workflow governance, because who reviews what stage is effectively part of access and responsibility management.

Key Technical Highlights

  • JWT decoding without an external auth SDK.
  • Context-based auth, notifications, and toasts instead of heavyweight global state.
  • Role-aware route protection and role-aware console behavior.
  • Reusable modal architecture for create, edit, and confirm-delete flows.
  • Singleton SSE notification stream to avoid duplicate real-time connections.

Challenges Solved

  • Preventing duplicate SSE connections across multiple notification consumers.
  • Supporting both admin and super admin workflows without duplicating all component logic.
  • Handling profile data, auth state, and user-admin operations without a large external state library.

Scalability Considerations

  • Add server-side pagination and filtering to admin user lists.
  • Introduce React Query or a similar cache layer for mutation invalidation and consistent loading states.
  • Split route bundles lazily for faster first load.

14. Client-Friendly Explanation

IdeaTrack is a web application that helps an organization collect ideas, review them properly, and turn strong ideas into formal proposals. Different people see different tools depending on their role. Employees submit ideas, reviewers assess them, and administrators manage users, categories, reporting, and system setup.

The User Management area lets administrators add new users, assign what kind of access they should have, update their information, and organize which reviewers handle which review stages. That makes the platform easier to operate and keeps responsibilities clear.

From a business perspective, the architecture is scalable because the application already separates authentication, user operations, profile management, notifications, and analytics into distinct modules. That means new roles, new screens, or deeper reporting can be added without redesigning the whole product.

15. Conclusion

IdeaTrack’s frontend is a well-structured React SPA built around role-based operations and a clear provider-driven shell. Its strongest architectural qualities are the separation between route orchestration, reusable feature components, and API utilities, plus a focused approach to global state.

The User Management module is technically significant because it connects access control, account lifecycle management, self-service profile operations, and reviewer assignment into one coherent operational capability. For an interview, it demonstrates practical frontend architecture, business workflow modeling, and real-world admin UX design. For a client, it demonstrates control, governance, and scalability.
